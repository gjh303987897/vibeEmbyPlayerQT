# 修复 WebDAV 上传取消后可能闪退

## 问题现象

用户反馈：在 WebDAV 中上传文件后点击“取消”，部分情况下可能出现程序闪退。

这类问题通常和异步网络请求、上传文件对象生命周期、取消后的回调顺序有关。

## 根因分析

### 1. 上传文件对象没有绑定到对应 reply 生命周期

旧实现中，上传文件用 `m_activeFile = std::make_unique<QFile>(...)` 保存。

上传开始时：

```cpp
reply = m_manager.put(request, m_activeFile.get());
```

取消时：

```cpp
m_activeReply->abort();
```

随后 finished 回调会进入 `finishActive(...)`，并关闭、释放 `m_activeFile`。

风险点在于：

- `QNetworkAccessManager::put(..., QIODevice*)` 会异步读取这个 `QFile`。
- 用户点击取消时，reply 和文件对象的销毁顺序要非常小心。
- 如果旧 reply 的信号在新任务启动后晚到，依赖全局 `m_activeFile` / `m_active` 容易误伤当前任务。

### 2. 回调通过当前全局 active 状态识别任务

旧的 `uploadProgress` / `downloadProgress` 回调通过 `m_active->task.id` 更新进度。

问题是取消后 `finishActive()` 会清理 active 并可能立即 `startNext()`，旧 reply 如果还有晚到信号，就可能错误使用新的 active 任务。

### 3. 排队任务取消时会重复插入任务记录

旧实现取消队列中的任务时：

```cpp
auto task = m_queue.takeAt(i);
task.task.status = statusCanceled();
m_tasks.push_back(task.task);
```

但该任务在入队时已经存在于 `m_tasks`，这里再次 push 会导致 UI 模型里出现重复任务。

### 4. 任务 ID 使用毫秒时间戳，批量上传存在碰撞风险

旧任务 ID：

```cpp
transfer-<currentMSecsSinceEpoch>
```

用户上传文件夹或多文件时，短时间内连续创建多个任务，有机会生成相同 ID。
取消按钮依赖 taskId，因此 ID 碰撞会让取消目标变得不可靠。

## 修复内容

### 1. 上传/下载文件对象绑定到 reply

将活动文件从：

```cpp
std::unique_ptr<QFile> m_activeFile;
```

改为：

```cpp
QPointer<QFile> m_activeFile;
```

上传或下载开始时：

- 使用 `new QFile(...)` 创建文件对象。
- 成功创建 reply 后调用 `file->setParent(reply)`。
- 让文件对象跟随对应 `QNetworkReply` 生命周期。

这样 reply 未完成前，上传所需的 `QIODevice` 不会被提前销毁；reply `deleteLater()` 时也会带走文件对象。

### 2. 每个 reply 保存自己的 taskId

开始传输后给 reply 设置：

```cpp
reply->setProperty("transferTaskId", m_active->task.id);
```

finished 回调中读取该 taskId：

```cpp
const auto taskId = reply->property("transferTaskId").toString();
```

`finishActive(...)` 也改为必须传入 taskId，并且只有当前 active 任务 ID 匹配时才会收尾。

这样旧 reply 的晚到信号不会错误结束新的 active 任务。

### 3. 进度更新按 taskId 定位任务

新增：

```cpp
updateProgress(const QString& taskId, qint64 done, qint64 total)
```

`uploadProgress` 和 `downloadProgress` 回调都捕获各自任务的 taskId，不再依赖当前 `m_active`。

下载 `readyRead` 回调也捕获对应的 `QFile*`，不再通过全局 `m_activeFile` 写入。

### 4. 修复排队任务取消重复插入

排队任务被取消时，不再向 `m_tasks` 追加新任务，而是在已有任务记录上更新：

- `status = canceled`
- `detail = Canceled`
- `cancellable = false`

并向外发出一致的 `Canceled` 消息。

### 5. 任务 ID 改为 UUID

任务 ID 从毫秒时间戳改为：

```cpp
transfer-<uuid>
```

避免批量上传或文件夹上传时多个任务在同一毫秒创建导致 ID 碰撞。


处理方式：

- 通过 `Get-Process vibePlayerQT` 找到进程。
- 使用 `Stop-Process` 结束该测试进程。
- 重新执行构建。

## 后续建议

如果后续还要增强传输稳定性，可以继续补一个独立的传输管理单元测试：

- 本地启动慢速 HTTP/WebDAV 接收端。
- 入队上传大文件。
- 上传开始后立即取消。
- 验证任务状态为 `canceled`，且不会启动错误任务或重复任务。
