# 修复 WebDAV 视频播放仍然 `Playback failed: loading failed`

## 问题现象

在 WebDAV 文件列表中点击视频播放，仍然出现：

```text
Playback failed: loading failed
```

用户确认同一视频在网页中通过 WebDAV 可以正常播放。

## 复查历史记录

修复前重新阅读了 `/bugFix/2026-06-22-14.26.39.md`。

上一次记录中的播放失败根因是 libmpv TLS 校验失败：

```text
[error/curl] TLS certificate verification failed.
```

这次没有直接沿用上一次结论，而是重新查看最新日志，确认新的失败原因已经变化。

## 新日志中的真实原因

最新日志显示：

```text
[error/curl] HTTP error 401
[error/stream] Failed to open https://...
```

因此这次不是 TLS 证书问题，而是 libmpv 直接访问 WebDAV 文件时没有通过认证。

## 原因分析

- WebDAV 目录浏览由 Qt 网络层完成。
- Qt 的 `QNetworkAccessManager` 可以通过 `authenticationRequired + QAuthenticator` 处理 Basic/Digest 认证。
- 但视频播放交给 libmpv 后，libmpv 不会复用 Qt 网络层的认证状态。
- 之前尝试在播放前设置 `http-user/http-password`，但复查 mpv 官方选项后发现这不是可靠的 mpv 网络认证方案。
- 对 rclone/OneDrive WebDAV 这类服务，远程视频请求最终返回 401，导致 libmpv 加载失败。

## 修复方案

改为新增本地 WebDAV 播放代理：

- 新增 `WebDavPlaybackProxy`。
- libmpv 不再直接访问远程 WebDAV 视频 URL。
- WebDAV 视频播放时，`AppViewModel` 生成一个 `http://127.0.0.1:<port>/<streamId>/stream` 本地 URL 给 libmpv。
- 本地代理收到 libmpv 请求后，使用 Qt 网络层访问真正的 WebDAV 远程视频 URL。
- 代理转发 `Range` 请求头，支持视频 seek 和分段读取。
- 代理继续使用 `QAuthenticator` 处理 WebDAV Basic/Digest 认证。
- 代理继续复用 WebDAV 服务的自签名证书设置。
- 播放结束或切换播放时撤销对应 streamId，避免无效本地 URL 长期存在。

## 修改内容

- 新增：
  - `src/services/webdav/WebDavPlaybackProxy.h`
  - `src/services/webdav/WebDavPlaybackProxy.cpp`
- 更新 `CMakeLists.txt`，加入新的代理模块。
- 更新 `AppViewModel`：
  - WebDAV 视频播放时创建本地代理 URL。
  - 保存当前代理 streamId。
  - 清理播放状态时撤销代理 stream。
- 保留上一次的 TLS 处理，但 WebDAV 远程认证现在由本地代理的 Qt 网络层处理。

## 后续注意

- 如果后续仍播放失败，应优先查看日志中本地代理请求远程 WebDAV 时的 HTTP 状态码。
- 不要再把所有 `Playback failed: loading failed` 直接归因于 TLS；必须先看 mpv/curl 的下一行错误。
