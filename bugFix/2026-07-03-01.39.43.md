# 修复历史统计中视频流量严重偏低

## 问题现象

用户播放在线视频并拖动进度条后，Windows 任务管理器显示实际网络流量已经达到数百 MB，但历史统计页面只记录到约 244.2 KB。

## 原因分析

旧实现只统计了以下网络入口：

- Emby / Jellyfin 的 Qt `NetworkClient` API 请求与响应。
- WebDAV 目录请求、上传下载任务。
- WebDAV 播放代理转发远端视频响应时收到的数据块。

但 Emby、Jellyfin 和 IPTV 的正片播放 URL 是直接交给 libmpv 的，真实的视频流下载、seek 后的 Range 请求或 HLS 分片请求都由 libmpv 自己完成，不经过 `QNetworkAccessManager`。

因此历史统计里只能看到登录、媒体详情、播放进度上报等少量 API 流量，无法看到视频本体流量，最终就会出现 KB 级统计值。

## 官方文档复查

复查了 mpv 官方手册：

- `cache-speed` 表示 cache 与底层输入层之间的当前 I/O 读取速度，单位为 bytes/s，client API 类型是 `MPV_FORMAT_INT64`。
- 该值等同于 `demuxer-cache-state/raw-input-rate`。
- `raw-input-rate` 是网络层或其他字节输入层的估算输入速率，可能不精确或缺失。

参考：

- https://mpv.io/manual/stable/

## 修复方案

新增播放器层流量采样：

- `PlayerController` 增加 1 秒定时器。
- `loadfile` 请求成功后启用采样，停止播放、播放结束和销毁时关闭采样。
- 定时读取 mpv 的 `cache-speed` 属性。
- 根据实际计时间隔把 bytes/s 换算为本次采样字节数。
- 通过 `playbackNetworkBytes(qint64 bytesReceived)` 信号向上转发。

新增转发链路：

- `MpvVideoItem` 转发 `PlayerController::playbackNetworkBytes`。
- `qml/Main.qml` 的播放器实例将采样结果传给 `AppViewModel::recordPlaybackNetworkBytes()`。
- `AppViewModel` 复用已有 `recordNetworkUsage()` 写入历史统计，因此隐私模式、按天聚合、刷新历史页和 30 天保留策略保持一致。

## WebDAV 去重

WebDAV 视频播放已经通过 `WebDavPlaybackProxy` 统计远端响应数据块。

libmpv 播放 WebDAV 时读取的是本地 `127.0.0.1` 代理 URL，如果同时统计 mpv `cache-speed`，会把本机回环读取也算进去，造成重复统计。

因此 `AppViewModel::recordPlaybackNetworkBytes()` 对 WebDAV 服务跳过播放器层采样，WebDAV 仍由代理负责统计真实远端流量。

## 没有改动的内容

本次没有修改：

- `PlayerController::playUrl()` 的 `loadfile` 参数顺序。
- Emby / Jellyfin / IPTV 播放 URL 生成逻辑。
- WebDAV 本地播放代理、认证转发和 Range 请求处理。
- Windows native mpv popup HWND 的创建、DPR 换算、geometry 同步和 Z 序逻辑。
- 播放器控制层与缓冲提示窗口层级方案。

## 注意

mpv 官方文档说明 `raw-input-rate` / `cache-speed` 是估算值，可能不精确或缺失。

因此历史统计中的视频流量目标是接近真实拉流量级，尤其覆盖 seek 触发的大量下载；它不等同于操作系统任务管理器统计的网卡字节数，任务管理器还可能包含协议开销、其他进程流量或本机回环差异。

## 验证

需要重点验证：

- 播放 Emby / Jellyfin / IPTV 在线视频时，历史统计流量随播放和 seek 明显增长。
- WebDAV 在线播放不会因为播放器层采样而重复统计。
- 隐私模式下播放产生的流量仍按隐私记录写入，非隐私模式历史页不显示隐私统计。
- 本地视频播放不应产生服务流量统计。
- 播放器窗口嵌入、seek、字幕、音轨切换和控制层显示不受影响。
